\section{Question One}

An \textit{esoteric programming language}, or \textit{'esolang'}, refers to a type of programming language designed to be unconventional and unlike traditional programming languages. They feature purposefully obfuscated syntax, unique compilation/interpretation processes, and generally aim to push the boundaries of language design into the realm of art. 
\section{Question Two}

\textit{Esolangs} have limited application in the same context as traditional programming languages, but that is not to assert that they have no practical applications.

\begin{enumerate}
    \item \textbf{Educational Value}
    \newline By stripping languages down to their bare essentials, they can aid in understanding memory models, compilation, and control flow with little to nothing in the way, unlike traditional languages \cite{singer2025esolang}.
    
    \item \textbf{Artistic Expression}
    \newline Some \textit{esolangs} treat code as a performative or artistic work, such as \textit{Piet} \citep{piet}, a visual language in which programs look like abstract, cubist paintings.
    
    \item \textbf{Security Potential}
    \newline Due to inherent code obfuscation, some languages like \textit{JSFuck} \citep{jsfuck} easily get past traditional malware detection. The opposite can also be done, sensitive code could theoretically have a built-in layer of 'encryption' by utilizing them.

    \item \textbf{Compiler/ Interpreter Research}
    \newline Some \textit{esolangs} feature unique compilation and interpretation processes. These can help contribute to understanding of the compilation process, and provide insights from a fresh angle. One example of this is \textit{Brainfuck} \citep{brainfuck}, which was designed to have the smallest possible compiler for the \textit{AmigaOS}.
    
\end{enumerate}

\section{Question Three}



\begin{enumerate}
    \item \textbf{Piet}
    \newline \citep{piet} is an \textit{esolang} designed by Ian Bogost in 2001. It is a visual language in which programs are represented as images composed of coloured blocks, called \textit{codels}. Program execution is determined by a pointer moving through these blocks.

    The semantics of the language are defined by changes in hue and brightness between adjacent codels. This corresponds to typical language operations like stack manipulation and i/o. For instance, a +1 in hue and lightness corresponds to 'push'.

\section{Question Four}

SALT was developed by researchers in formal methods at various European and Australian institutions \citep{salt}. \newline It was not created as a general purpose programming language, rather being a specification and assertion language based on \textbf{temporal logic}. This allows developers to formally describe the behaviours and properties of their systems over time, especially useful for reactive and concurrent systems.

The main advantage of SALT lies in its rigor. The use of a mathematically grounded method of stating and verifying properties, significantly reduces the ambiguity at runtime, and supports formal verification that would be impossible to achieve through testing alone.

Two real world domains where SALT would be applicable:
 \newline - Safety-critical systems, where we want extreme correctness and predictability, like medical devices.
 \newline - Concurrent and distributed systems, where interaction between components may get messy without proper rules.

\section{Question Five}

Vibe coding is a term described by Andrej Kaparthy \citep{karpathy_vibe} as a programming apporach where the 'developer' relies heavily on intution, experimentation, and especiallly AI generated code, rather than going through a traditional planning and iterative approach.

\textbf{Potential advantages:}

\newline - Increased productivity - The raw code output of a vibe coder is far higher than that of a traditional programmer, as AI tools can rapidly generate boilerplate and simple code
\newline - Low barrier to entry - Vibe coding is possible for anyone, allowing users to build systems without any in depth knowledge. Vibe coding may serve as a learning tool, allowing those with interest to get involved in software development

\textbf{Potential drawbacks:}

\newline Technical debt - Although a system may work, when limited planning has gone in to it, edge cases, performance issues, reduced scalability and security concerns become an issue, possibly leading to a higher cost than designing a system traditionally from the start.

\newline Over-reliance on tooling - Dependence on AI tools is becoming an increasing concern, with developers struggling to reason independently.
